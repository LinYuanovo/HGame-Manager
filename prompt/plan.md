# 项目迁移提示词：从 JAV-Manager 改造为本地游戏管理工具

## 一、原始项目说明
原始项目 `JAV-Manager` 是一个基于 Flutter 开发的 Windows 本地视频媒体管理器，具有以下特点：
- 侧边栏：刮削中心、媒体库、演员、分类、收藏、已观看
- 刮削功能通过调用独立的 Python 子进程（`javsp_scraper.exe`）实现
- 支持自动整理媒体文件（移动已观看影片到指定目录）
- 内置视频播放相关功能
- 数据模型基于影片（番号、演员、标签、系列、片商等）
- 界面风格：玻璃拟态、现代化 UI、多种视图模式（列表/海报/海报墙等）

详细代码结构见 README（附件已提供）。

## 二、迁移目标
从上述项目改造出一个**本地游戏管理工具**，保留原有 UI 风格和基础架构，删除/修改不相关功能，增加游戏管理所需的能力。

### 目标平台
Windows 10/11（64位），继续使用 Flutter + Dart。

## 三、总体修改清单

### 3.1 导航侧边栏（修改）
原项 → 新项：
- 刮削中心 → **刮削**（保留，但实现方式改为纯 Dart）
- 媒体库 → **游戏**（重命名，内容逻辑变化）
- 演员 → **删除**（不需要演员管理）
- 分类 → **分类**（保留，但内容改为标签/系列，系列预定义类型）
- 收藏 → **收藏**（保留，但增加“游戏”和“分类”两个 tab）
- 已观看 → **已玩**（重命名，追踪游戏游玩状态）

### 3.2 刮削功能（重写）
- **删除** Python 刮削器调用，改为 **纯 Dart 实现**
- 用户提供一个 `web/` 目录，里面存放多个不同游戏网站的 HTML 示例文件
- 扫描本地游戏文件夹：若文件夹内包含 `source_url.txt`，则认为该文件夹是一个游戏
- 刮削流程：
  1. 读取 `source_url.txt` 中的 URL
  2. 使用 Dart 的 `http` 包请求该 URL（支持代理设置）
  3. 根据 URL 域名匹配对应的解析器（从 `web/` 中的 HTML 样本逆向编写解析逻辑）
  4. 解析出元数据（标题、版本、介绍、配图链接、游戏特点、更新内容、下载链接、出处链接等）
  5. 生成 `metadata.json` 或类似的元数据文件保存在游戏文件夹内
  6. 支持用户**手动编辑元数据**并保存（UI 上提供编辑表单）

### 3.3 删除的功能模块
- **播放相关**：删除所有视频播放器、播放历史、播放进度相关代码
- **自动整理**：删除“已观看后自动移动到指定目录”的逻辑
- **演员管理**：删除所有演员页面、演员头像获取、演员详情页
- **NFO 文件依赖**：不再依赖 `movie.nfo`，改用新的元数据结构（JSON）
- **整理完成目录**：不再要求 `#整理完成/` 这种结构

### 3.4 游戏页面（原媒体库页面）
- 展示所有扫描到的游戏（每个游戏文件夹视为一个游戏项）
- **视图模式**：仅保留 **列表模式** 和 **海报墙模式**，删除其他视图
- **分页**：支持瀑布流加载（无限滚动）或传统的**分页控件**（每页 N 条，用户可配置）
- **排序**：按标题、最近游玩（已玩时间）、添加时间、自定义标签等
- **右键菜单**（`context menu`）：
  - 打开文件夹（在资源管理器中打开游戏所在目录）
  - 收藏/取消收藏
  - 标记已游玩（记录游玩时间、次数）
  - 添加自定义标签
  - 移除媒体库（从数据库删除，不删除文件）
  - 删除本地文件（永久删除文件夹及内容，需二次确认）

### 3.5 游戏详情页面（完全重新设计）
- 页面可上下滑动（类似文章详情页），不再是原项目的左右分栏
- 展示内容字段（需对应数据库字段）：
  - **标题**（string）
  - **版本**（string）
  - **标签**（list<string>，包含预定义系列标签和自定义标签）
  - **游戏路径**（string，本地文件夹绝对路径）
  - **游戏介绍**（长文本）
  - **游戏配图**：从游戏文件夹下的 `image/` 子目录中读取所有图片（支持 jpg/png/webp），作为轮播图或相册展示
  - **游戏特点**（长文本）
  - **更新内容**（长文本，最近更新说明）
  - **下载链接**（string，原始下载 URL）
  - **出处链接**（string，来源页面 URL，即 `source_url.txt` 中的地址）
- 提供**编辑按钮**：允许用户修改上述所有可编辑字段（除路径外），保存后更新数据库和本地元数据文件

### 3.6 分类页面（修改）
改为两个 Tab：
- **标签**（Tab 1）：
  - 展示所有标签（包括预定义系列标签 + 用户自定义标签）
  - 点击标签显示包含该标签的游戏列表
  - 支持添加/删除自定义标签（标签全局管理）
- **系列**（Tab 2）：
  - 系列是预定义的游戏类型，至少包括：`RPG, ADV, ACT, SLG, AVG, FPS, TPS`
  - 每个系列可以有一个中文显示名（例如 RPG -> 角色扮演）
  - 点击系列显示该类型下的游戏列表
  - 系列标签可由用户自定义扩展（可选需求）

注意：原有的“片商”、“演员”等分类全部删除。

### 3.7 收藏页面（修改）
改为两个 Tab：
- **游戏**：展示用户收藏的游戏列表（支持搜索、排序、移除收藏）
- **分类**：展示用户收藏的标签/系列（支持快速跳转到该分类下的游戏列表）

### 3.8 已玩页面（原已观看页面）
- 记录用户标记为“已游玩”的游戏
- 显示游玩次数、最后游玩时间
- 支持取消标记（从未玩列表中移除）
- 不实现自动移动文件功能

### 3.9 数据库设计（需调整）
保留原有的 SQLite 架构，但替换表结构。建议设计以下核心表：
- `games`：游戏主表（id, title, version, path, intro, features, changelog, download_url, source_url, added_time, last_played_time, play_count, is_favorite, ...）
- `game_images`：游戏配图表（game_id, image_path, sort_order）
- `tags`：标签表（id, name, type( 'custom' | 'series' ), display_name）
- `game_tag_relation`：游戏与标签的关联表
- `settings`：应用设置（媒体库根目录、代理、分页大小等）

### 3.10 扫描媒体库逻辑修改
- 监听用户设置的“游戏库根目录”
- 递归扫描所有子文件夹，若文件夹包含 `source_url.txt` 则识别为一个游戏
- 读取文件夹下的 `metadata.json`（如果存在）加载元数据，否则仅显示基础信息（文件夹名作为标题）
- 支持手动触发扫描、自动扫描（文件系统监控可选）

## 四、保留的原有功能/组件
- 玻璃拟态 UI 主题（暗色/亮色模式）
- 自适应网格布局、列表布局组件（可复用）
- 窗口管理（位置、尺寸持久化）
- Riverpod 状态管理架构
- 通用的 HTTP 请求工具（需扩展支持代理）
- 收藏、标记状态的本地存储模式
- 右键菜单的实现方式
- 设置页面（修改媒体库路径、代理、分页大小等）

## 五、技术实现要点
1. **刮削器实现**：
   - 在 `lib/scraper/` 下新建 `html_parser.dart`，针对不同网站实现 `SiteParser` 抽象类
   - 从 `web/` 目录中读取 HTML 样本，分析 CSS 选择器或正则提取数据
   - 支持动态 JS 渲染？优先使用简单 HTTP 请求 + 解析 HTML，若需要可集成 `webview` 或 `puppeteer`（但会增加复杂度，先不考虑）
2. **图片处理**：游戏配图从 `游戏文件夹/image/` 加载，使用 `Image.file` 或缓存
3. **文件操作**：删除本地文件需使用 `Directory.delete` 并处理权限
4. **右键菜单**：使用 `PopupMenuButton` 或 `context_menu` 包实现
5. **分页/瀑布流**：使用 `ListView.builder` + 分页加载数据（数据库 `LIMIT/OFFSET`）
6. **游戏详情页面**：使用 `SingleChildScrollView` + `Column` 布局，自定义卡片样式
